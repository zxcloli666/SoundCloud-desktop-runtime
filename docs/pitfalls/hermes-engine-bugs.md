# Hermes engine bugs (не наш код)

Баги в самом движке Hermes, не в нашем коде — сюда попадает только то, что
подтверждено изолированным репродюсом (и, где возможно, сравнением с V8/Node
на той же самой сборке).

## `for (let key of ...)` не даёт свежий биндинг на каждой итерации

**Симптом**: любое свойство CJS-модуля, обёрнутого esbuild-овским
`__copyProps` (CJS→ESM interop для именованных импортов), начинало
возвращать ПОСЛЕДНЕЕ enumerable-свойство модуля вместо реального значения.
Конкретно: `import { createContext, useMemo } from 'react'` → и
`createContext`, и `useMemo` превращались в строку `"19.2.3"` (react's
`version`) → `'19.2.3' is not a function`.

**Причина**: `for (let key of __getOwnPropNames(from))`, где тело цикла
создаёт замыкание через `Object.defineProperty(to, key, { get: () =>
from[key], ... })` — в Hermes `key` НЕ получает свежий per-iteration
биндинг, все геттеры в итоге видят один и тот же (последний) `key`. Это
ТОЧНАЯ форма esbuild-овского `__copyProps` helper'а. Подтверждено:
изолированный репро вне бандла (`crates/js-host/src/tests/
hermes_engine_bug.rs`) воспроизводит баг сам по себе; тот же JS в Node
(V8) выполняется верно — баг специфичен для Hermes.

**Почему не ES5-таргет**: `for...of`+`let` — ES2015; понижение ВСЕГО
бандла до ES5 не подошло — проект не транспилится целиком (const/
деструктуризация ломаются на других местах).

**Фикс**: постпроцессинг бандла (`js/build-support.mjs`'s
`patchHermesForOfBug`, зовётся из каждого `build.mjs`) — regex-патч меняет
`for...of`+`let` на `.forEach(function(key) {...})` (параметр функции —
Hermes с ним работает верно, в отличие от `let` в цикле). Патч кидает
ошибку сборки, если esbuild когда-нибудь поменяет форму helper'а (нужно
будет обновить regex) — для бандлов, где патч заведомо не обязателен
(движковый `js/playground/`, у которого нет именованных CJS-импортов),
вызывается с `required: false` — тихий no-op, если паттерн не найден.
