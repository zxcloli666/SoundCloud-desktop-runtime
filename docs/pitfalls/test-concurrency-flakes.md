# Тесты: process-global каналы, дренящиеся полностью на каждый вызов

Один и тот же класс бага, найден дважды в разных модулях — оба раза
из-за того, что фоновая доставка результата (mpsc-канал, задренированный
раз в кадр) — процесс-глобальное состояние, а `cargo test` по умолчанию
гоняет каждый `#[test]` на своём OS-потоке.

## `live_data_test`: коллизия callback_id

**Симптом**: тест, ожидающий свой собственный `sc_rn::auth_status()`
результат, изредка получал результат ОТ ДРУГОГО теста/бандла.

**Причина**: `async_bridge`'s tokio-рантайм + mpsc-канал — один
процесс-глобальный инстанс, общий для КАЖДОГО `Runtime::new()` в тестовом
бинарнике. Бандл-тесты (`bundle_test`/`reanimated_test`/
`fills_arbitrary_aspect_ratio_test`, каждый — отдельный Hermes-рантайм) в
какой-то момент тоже гоняли `LiveDataProbe`'s `authStatus()` с
callback_id, начинающимся с 1 — маленький хардкоженный id в
`live_data_test` мог получить чужой, ещё не забранный результат.

**Фикс**: различимые, заведомо вне обычного диапазона callback_id
(100001+).

## `image_cache_test`: кросс-поточная кража результата (различимые id НЕ спасли)

**Симптом**: 1-2 из 3 параллельных `#[test]` изредка падали с "did not
complete within 5s", хотя их фетч объективно успевал.

**Причина**: НЕ коллизия id на этот раз — `drain_ready()` полностью
опустошает ОБЩИЙ mpsc-канал на каждый вызов; три `#[test]`-функции,
каждая на своём потоке, поллили этот канал независимо. Один поток мог
вычерпать и молча выкинуть готовый результат ДРУГОГО, ещё выполняющегося
теста (его `wait_for_result`-хелпер проверял только свой собственный id,
не совпавшие результаты просто отбрасывались). Кросс-поточный "стэш
неразобранных результатов" тоже не вариант — `skia_safe::Image`
(`RCHandle<SkImage>`, голый `NonNull`) не `Send`/`Sync`, никакого
`unsafe impl` для него в `skia-safe` нет.

**Фикс**: слить все три сценария (реальный URL, битый URL, дубликат
запроса) в ОДИН `#[test]`, весь polling на одном потоке — красть
физически нечему, никакого шаренного канала между тестами.

**Урок на будущее**: если тест поллит process-global "дренится-целиком"
канал/реестр — считать это потенциальным источником кросс-теста
interference С САМОГО начала, не только когда флейк уже проявился.
Production-код (единственный поток рендер-лупа) этим классом бага не
страдает — риск чисто тестовый.
